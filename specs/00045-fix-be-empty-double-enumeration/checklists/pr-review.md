# PR Review Checklist: Fix BeEmpty Double Enumeration

**Purpose**: Validate the requirement quality, clarity, and completeness of the BeEmpty double-enumeration fix before merging — not to verify the implementation, but to confirm the spec is well-written and reviewer-ready.
**Created**: 2026-06-09
**Feature**: [spec.md](../spec.md) | [plan.md](../plan.md) | [contracts/public-api.md](../contracts/public-api.md)
**Depth**: Standard review gate | **Audience**: PR reviewer

## Requirement Completeness

- [x] CHK001 — Are requirements defined for all three subject states: null, empty, and non-empty? The null case appears only in Edge Cases and Assumptions, not in a formal FR. [Completeness, Gap, Spec §Assumptions] *(Null state explicitly scoped out in Assumptions: "Non-null subject handling remains unchanged." Edge Cases confirms: "existing null-handling behavior is preserved." Deliberate exclusion, properly documented.)*
- [x] CHK002 — Is the `AndConstraint` return-value behavior on the passing path documented in the spec or contract? No FR covers what is returned when the assertion passes. [Completeness, Gap, Spec §FR-001] *(contracts/public-api.md documents "Returns `AndConstraint` for chaining." FR-004 preserves all existing behavior including return type. Out of scope for a targeted bug fix.)*
- [x] CHK003 — Is the `because` / `becauseArgs` parameter behavior specified for the non-replayable failure case? Existing FRs do not address whether the custom reason is included in the failure message. [Completeness, Gap, Spec §FR-002] *(The `because` parameter is passed directly to `MessageFormatter.Fail()` — unchanged by the fix. FR-004 preserves existing behavior. An existing test `BeEmpty_WithBecauseReason_IncludesReasonInMessage` covers this path and must pass per FR-004.)*
- [x] CHK004 — Is there a formal requirement covering what happens when `Count()` throws during the single enumeration? Edge Cases notes the exception "propagates naturally" but no FR codifies this. [Completeness, Gap, Spec §Edge Cases] *(Edge Cases: "The exception propagates naturally — this fix does not add catch blocks." Unambiguous statement. No FR needed for a no-op behavior.)*

## Requirement Clarity

- [x] CHK005 — Is "non-replayable" defined or linked to a reference definition in the spec? The term is central to the defect description but is not formally defined anywhere in the document. [Clarity, Ambiguity, Spec §User Story 1] *(Three concrete examples given: "LINQ-to-database queries, Channel<T> readers, or custom iterators with side effects." Sufficient for any .NET developer. No formal definition needed for a targeted fix.)*
- [x] CHK006 — Is "exactly once" in FR-001 expressed in a way that is objectively testable without inspecting internal enumerator calls? The spec relies on the test helper (`ThrowOnSecondEnumerationSequence`) in the quickstart — is this testing strategy referenced in the spec itself? [Clarity, Spec §FR-001] *(US1 Independent Test field references the ThrowOnSecondEnumerationSequence approach. quickstart.md documents the full helper. The testing strategy is traceable from the spec through quickstart.md.)*
- [x] CHK007 — Is "on the happy path" in FR-005 defined precisely? It is unclear whether "happy path" means "assertion passes (empty collection)" or "no exception of any kind". [Clarity, Ambiguity, Spec §FR-005] *(RESOLVED by remediation: FR-005 now reads "When the assertion passes (subject is empty and no exception is thrown).")*
- [x] CHK008 — Is "additional allocations or enumerations" in FR-005 scoped relative to a defined baseline? Without a baseline, it is not possible to objectively verify whether the fix introduces new allocations. [Clarity, Spec §FR-005] *(RESOLVED by remediation: FR-005 now references "the constitution §VI.2 limit of a single enumerator and the result object" as the explicit baseline.)*

## Requirement Consistency

- [x] CHK009 — Are FR-004 ("all passing tests MUST continue to pass") and SC-003 ("all existing BeEmpty tests pass without modification") consistent and non-redundant? If both exist, is the distinction meaningful? [Consistency, Spec §FR-004, §SC-003] *(RESOLVED by remediation: FR-004 now constrains implementation approach — "no modification of any existing test file." SC-003 remains the observable outcome. Distinct and non-redundant.)*
- [x] CHK010 — Does the single-enumeration guarantee in FR-001 align with the allocation constraint in FR-005? For an empty sequence, `Count()` enumerates once and returns 0 — no failure path is entered. Is this case (0 allocations, 1 enumeration) explicitly reconciled? [Consistency, Spec §FR-001, §FR-005] *(RESOLVED by remediation: FR-005 explicitly covers "When the assertion passes (subject is empty)" — 1 enumeration, 0 new allocations. The empty-sequence case is now explicitly reconciled.)*
- [x] CHK011 — Are the failure-message examples in User Story 1 and SC-001 consistent? Both use "a collection with N item(s)" — confirm the format string is identical across all spec sections. [Consistency, Spec §User Story 1, §SC-001] *(US1 scenario 1: "a collection with 3 item(s)". SC-001: "a collection with N item(s)". FR-003 formalizes the format. Consistent across all sections.)*

## Acceptance Criteria Quality

- [x] CHK012 — Is SC-001 measurable independently of knowing the element count before the assertion? The criterion requires the message to contain the count of the actual sequence — is the test scenario (3 items → "3 item(s)") the only concrete example, or are range values also in scope? [Acceptance Criteria Quality, Spec §SC-001] *(SC-001: "where N matches the actual element count." The 3-item example is a specific test scenario; the criterion applies for any N. Measurable for all non-empty inputs.)*
- [x] CHK013 — Is SC-002 ("does not throw a secondary exception") specific enough? It does not state which exception type would indicate a regression (`InvalidOperationException`, `ObjectDisposedException`, etc.). [Clarity, Spec §SC-002] *(Any uncaught exception during failure-message construction is a regression regardless of type. The untyped criterion is intentionally broad and correct. The ThrowOnSecondEnumerationSequence test helper makes the scenario concrete.)*
- [x] CHK014 — Is SC-004 ("BeEmpty on an empty non-replayable sequence passes") independently testable? Is a concrete test scenario documented that exercises this path? [Acceptance Criteria Quality, Spec §SC-004] *(T004 (`BeEmpty_WithNonReplayableEmptySequence_Passes`) directly exercises this. quickstart.md documents the full scenario including the helper class.)*
- [x] CHK015 — Can all four SC items be objectively verified by a reviewer without knowledge of the implementation? Specifically SC-002 and SC-004 rely on the absence of an event — are these absence-of-failure criteria considered sufficient without a positive signal? [Measurability, Spec §SC-002, §SC-004] *(Absence-of-exception assertions are standard and unambiguous in xUnit. SC-002 is verified by T003 (exception would propagate if fix is wrong). SC-004 by T004 (assertion failure would throw if fix is wrong). Both are positively falsifiable.)*

## Scenario Coverage

- [x] CHK016 — Are requirements defined for the chaining scenario (`BeEmpty().And.HaveCount(0)`)? The fix changes how `BeEmpty` exits — is the fluent chain contract preserved under the fix? [Coverage, Gap] *(The fix does not change the return statement — `return new AndConstraint<GenericCollectionAssertions<T>>(this)` is unmodified. FR-004 (no existing test modification) implicitly preserves chaining. contracts/public-api.md documents the return type.)*
- [x] CHK017 — Is the concurrent-call scenario explicitly excluded or addressed? The constitution (§3.3) requires assertion methods to be safe for concurrent use — the spec does not reference this. [Coverage, Gap, Constitution §3.3] *(The fix introduces `var count` as a stack-local variable — inherently thread-safe, no shared mutable state introduced. Constitution §3.3 compliance is maintained by design; no spec update needed for a no-new-state change.)*
- [x] CHK018 — Are requirements defined for sequences that yield elements lazily with observable side effects (not just throwing on second enumeration)? The spec example covers the throw case but not the silent-wrong-count case. [Coverage, Spec §User Story 1] *(FR-001 covers all non-replayable sequences: "regardless of whether the subject is a replayable or non-replayable `IEnumerable<T>`." The single-Count() fix is correct for all variants — LINQ queries, Channel readers, side-effecting iterators — whether they throw, return 0, or return a different count on replay.)*
